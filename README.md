# 배송 현황 카드 컴포넌트

## 1. 프로젝트 소개

물류 운영팀 내부 대시보드에서 하루 수백~수천 건의 배송 건을 빠르게 모니터링하기 위한 React + TypeScript 기반 배송 현황 카드 컴포넌트입니다.

배송 상태별 시각 구분, 예외 상태 강조, 진행 타임라인, 상태 필터, 운송장번호 검색, 페이지네이션, 상세 모달을 구현했습니다.

## 2. 실행 방법

```bash
npm install
npm run dev
```

빌드 확인:

```bash
npm run build
```

## 3. 구현 범위

- 배송 준비 중, 배송 중, 배송 완료, 배송 지연, 반송 5가지 상태 표현
- 송장번호, 수령인, 출발지, 도착지, 예상 도착일 표시
- 상태별 색상, 아이콘, 배지 적용
- 배송 준비 → 집화 → 이동 중 → 배송 완료 타임라인 UI
- 배송 지연 상태 warning strip 표시
- 카드 클릭 시 상세 모달 표시
- 상세 모달 내 보낸 사람, 수령인, 배송 담당자 정보 표시
- 배송 이동 이력 및 최근 업데이트 시간 표시
- 전체 / 준비중 / 배송중 / 완료 / 지연 / 반송 상태 필터
- 운송장번호 검색 및 초기화
- 페이지네이션
- 초기 로딩 Skeleton UI
- PC 2열 / 모바일 1열 반응형 레이아웃
- 150건 mock 데이터와 페이지당 10건 목록 표시

## 4. 디자인 결정 사항

### 정보 우선순위

목록 카드에서는 운영자가 빠르게 확인해야 하는 정보만 노출했습니다.

- 송장번호
- 배송 상태
- 수령인
- 출발지 / 도착지
- 예상 도착일
- 진행 타임라인

보낸 사람 연락처, 수령인 연락처, 배송 담당자 연락처처럼 상세 확인 시 필요한 정보는 목록에 노출하지 않고 상세 모달에서만 확인할 수 있도록 했습니다. 목록의 정보량을 줄이고 여러 배송 건을 빠르게 훑을 수 있게 하기 위한 선택입니다.

### 상태별 시각 구분

배송 상태는 운영자가 가장 빠르게 확인해야 하는 정보이므로 카드 상단에 상태 배지를 배치했습니다. 상태별 색상은 CSS 변수와 상태 config로 관리해 카드, 필터, 모달에서 일관되게 사용할 수 있도록 했습니다.

배송 지연은 운영자가 우선 확인해야 하는 예외 상태라고 판단해 카드 상단에 warning strip을 추가했습니다. 반송은 이미 예외 처리 흐름에 들어간 상태로 보고, 별도 warning strip 없이 상태 색상과 배지로 구분했습니다.

### 타임라인 UI

배송 준비 → 집화 → 이동 중 → 배송 완료는 물류 업무에서 자연스러운 선형 흐름이라고 판단했습니다. 완료된 단계는 활성 색상으로 표시하고, 아직 진행되지 않은 단계는 비활성 스타일로 구분했습니다. 지연/반송 상태에서는 예외 아이콘을 사용해 정상 흐름과 문제 상태를 함께 파악할 수 있도록 했습니다.

### 레이아웃

PC에서는 2열 카드 레이아웃을 사용했습니다. 3열은 카드 폭이 좁아져 송장번호, 주소, 타임라인의 가독성이 떨어질 수 있고, 1열은 한 화면에서 확인 가능한 배송 건 수가 적다고 판단했습니다. 모바일에서는 1열로 전환해 좁은 화면에서도 정보가 깨지지 않도록 했습니다.

출발지와 도착지는 하나의 배송 경로 정보로 보고 같은 영역 안에 묶었습니다. 두 정보를 가깝게 배치해 사용자가 이동 경로를 한 덩어리로 스캔할 수 있도록 했습니다.

## 5. 트레이드오프

현재 페이지네이션은 mock 데이터 기준의 클라이언트 페이지네이션입니다. 실제 운영 환경에서는 API의 `page`, `pageSize`, `status`, `invoiceNo` query와 연결해 서버 페이지네이션으로 확장하는 것을 전제로 설계했습니다.

시간 제약상 아래 기능은 구현하지 않았습니다.

- 실제 API 연동
- 실시간 배송 상태 업데이트
- 서버 페이지네이션 또는 가상 스크롤
- 스크롤이 길어졌을 때 필터와 검색 영역을 상단에 고정하는 sticky 처리
- 페이지 수가 많아졌을 때 원하는 페이지로 바로 이동하는 페이지 jump 기능
- 담당자 배정, 알림, 카드별 액션 메뉴
- 실제 전화번호 마스킹 또는 권한별 개인정보 노출 제어
- 접근성 자동 테스트와 E2E 테스트

### 개선 방향

- 실제 API 응답 스키마와 연결
- 서버 페이지네이션, 캐싱, windowing 적용
- 필터/검색 영역 sticky 처리
- 페이지 번호가 많아질 경우 ellipsis 처리 또는 페이지 직접 입력 기능 추가
- 지연 사유, SLA 초과 시간, 담당 센터 필터 추가
- 담당자 배정 및 내부 메모 수정 기능 추가
- 개인정보성 정보에 대한 권한별 노출 제어
- 접근성 테스트 및 E2E 테스트 추가

## 6. 어려웠던 점

상태별 UI를 일관되게 유지하면서도 지연과 반송 같은 예외 상태를 어떻게 구분해 보여줄지 고민했습니다. 지연은 즉시 확인이 필요한 운영 경고로 보고 강하게 표시했고, 반송은 상태 색상과 배지로 구분했습니다.

또한 긴 수령인 이름과 긴 주소가 카드 레이아웃을 깨지 않도록 `overflow-wrap`을 적용했습니다. 빈 메모는 `등록된 메모가 없습니다.`라는 대체 문구를 제공해 정보 공백이 UI 오류처럼 보이지 않도록 처리했습니다.

Mock 데이터는 5가지 상태가 고르게 노출되도록 구성했고, 긴 텍스트, 빈 메모, 지연/반송 이력, 상세 연락처 정보를 포함해 다양한 UI 상태를 확인할 수 있도록 했습니다.
