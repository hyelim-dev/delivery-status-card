# 배송 현황 카드 컴포넌트

## 1. 프로젝트 소개

물류 운영팀 내부 대시보드에서 하루 수백~수천 건의 배송 건을 빠르게 모니터링하기 위한 React + TypeScript 기반 배송 현황 카드 컴포넌트입니다.

운영자가 목록을 빠르게 훑으며 배송 상태, 예외 상황, 현재 진행 단계를 인지할 수 있도록 상태별 색상/아이콘, 타임라인, 필터, 송장번호 검색, 페이지네이션, 상세 모달을 구현했습니다.

## 2. 실행 방법

```bash
npm install
npm run dev
```

빌드 확인:

```bash
npm run build
```

## 3. 구현 범위

- 배송 준비 중, 배송 중, 배송 완료, 배송 지연, 반송 5가지 상태 표현
- 송장번호, 수령인, 출발지, 도착지, 예상 도착일 표시
- 상태별 색상, 아이콘, 배지 적용
- 일반 배송 타임라인: 배송 준비 → 집화 → 이동 중 → 배송 완료
- 반송 타임라인: 반송 접수 → 회수 중 → 반송 완료
- 배송 지연 상태 강조 영역 표시
- 카드 클릭 시 상세 모달 표시
- 상세 모달 내 발송인, 수령인, 배송 담당자 정보 표시
- 배송 이동 이력, 최근 업데이트, 메모 표시
- 전체 / 준비중 / 배송중 / 완료 / 지연 / 반송 상태 필터
- 송장번호 검색 및 검색 초기화
- 페이지네이션
- 초기 로딩 스켈레톤 화면
- PC 2열 / 모바일 1열 반응형 레이아웃
- 150건 mock 데이터와 페이지당 10건 목록 표시
- 긴 수령인명, 긴 주소, 빈값/누락 데이터 예외 처리

## 4. 기술 스택

- React: 카드, 타임라인, 필터, 상세 모달을 컴포넌트 단위로 분리하기 위해 사용
- TypeScript: 배송 데이터와 상태값을 타입으로 관리해 렌더링 분기와 빈값 처리의 안정성을 높이기 위해 사용
- Vite: 빠른 개발 환경 구성을 위해 사용
- styled-components: 상태별 색상, 강조 영역, 반응형 스타일을 컴포넌트 단위로 관리하기 위해 사용
- lucide-react: 상태 배지, 타임라인, 검색, 페이지네이션, 모달 닫기 등에 필요한 SVG 아이콘을 일관된 스타일로 사용하기 위해 적용

## 5. 디자인 결정 사항

### 컴포넌트 구조 분리

대시보드 화면, 카드, 타임라인, 필터, 페이지네이션, 상세 모달의 역할을 분리했습니다.  
`DeliveryDashboard`는 필터/검색/페이지네이션 상태와 목록 계산을 담당하고, `DeliveryCard`는 카드의 정보 표시와 클릭 진입점을 담당합니다. `DeliveryTimeline`은 배송 단계 표현만 담당하도록 분리해 상태별 진행 UI를 독립적으로 관리했습니다.

이 구조는 카드 UI를 중심으로 평가하는 과제 특성에 맞춰 각 컴포넌트의 책임을 명확히 하고, 상태 추가나 모달 정보 변경 시 수정 범위를 줄이기 위한 선택입니다.

### 상태 관리 방식

전역 상태 관리 라이브러리 없이 `useState`, `useMemo`, `useCallback` 중심으로 구성했습니다. 현재 과제 범위에서는 필터, 검색어, 현재 페이지, 선택된 배송 건 정도만 관리하면 충분하다고 판단했습니다.

필터링된 목록과 페이지 단위 목록은 `useMemo`로 계산해 불필요한 재계산을 줄였고, 카드 클릭과 검색/초기화 같은 이벤트는 필요한 범위에서 `useCallback`을 사용했습니다. 과제 규모에 비해 과한 상태 관리 도구를 도입하지 않는 방향을 선택했습니다.

### 색상 및 아이콘

배송 상태는 운영자가 가장 먼저 구분해야 하는 정보이므로 상태별 색상을 분리했습니다. 준비중은 대기 상태에 가까워 `#64748B`를 사용했고, 배송중은 현재 진행 중인 상태를 나타내기 위해 `#2563EB`를 사용했습니다. 완료는 정상적으로 마무리된 상태를 표현하기 위해 `#16A34A`를 사용했습니다. 지연은 운영 확인이 필요한 경고 상태이므로 `#F59E0B`, 반송은 정상 배송 흐름에서 벗어난 상태이므로 `#E11D48`을 사용했습니다.

색상만으로 상태를 구분하지 않도록 아이콘과 텍스트를 함께 사용했습니다. 배송중은 차량 아이콘, 완료는 체크 아이콘, 지연은 경고 아이콘, 반송은 되돌아가는 방향의 아이콘을 사용해 상태 의미가 직관적으로 드러나도록 했습니다.

### 타이포그래피

폰트는 `Noto Sans KR`을 기본으로 사용하고, 대체 폰트로 `Pretendard`와 시스템 sans-serif 계열을 설정했습니다. 운영 대시보드에서는 송장번호, 이름, 주소, 날짜 정보의 판독성이 중요하다고 판단했습니다.

목록에서는 송장번호, 수령인, 상태 배지처럼 빠르게 확인해야 하는 정보에 무게를 두고, 주소와 상세 정보는 상대적으로 낮은 강조로 처리해 정보 위계가 흐려지지 않도록 했습니다.

### 정보 우선순위

목록 카드에는 운영자가 빠르게 판단해야 하는 핵심 정보만 노출했습니다.

- 배송 상태
- 송장번호
- 수령인
- 출발지 / 도착지
- 예상 도착일
- 진행 타임라인

발송인 연락처, 수령인 연락처, 배송 담당자 연락처처럼 개인정보성이 있거나 상세 확인 시 필요한 정보는 목록에 노출하지 않고 상세 모달에서만 확인하도록 했습니다. 목록은 빠른 스캔, 모달은 운영 대응에 필요한 상세 확인이라는 역할로 분리했습니다.

### 타임라인 UI

일반 배송은 `배송 준비 → 집화 → 이동 중 → 배송 완료`의 선형 흐름으로 표현했습니다. 완료된 단계는 채워진 노드와 체크 아이콘으로 표시하고, 현재 단계는 강조 노드, 미진행 단계는 외곽선 스타일로 구분했습니다.

반송은 정상 배송 흐름과 다르므로 `반송 접수 → 회수 중 → 반송 완료` 3단계로 별도 구성했습니다. 반송 상태가 배송 완료까지의 일반 흐름처럼 보이지 않도록 예외 흐름을 분리했습니다.

### 예외 상태 처리

배송 지연은 운영자가 우선 확인해야 하는 상태로 보고 카드 상단에 `배송 지연 · 운영 확인 필요` 강조 영역을 추가했습니다. 지연 상태는 `#F59E0B` 계열의 상단 강조선과 테두리를 함께 사용해 목록에서 빠르게 눈에 들어오도록 했습니다.

반송은 별도의 경고 띠를 사용하지 않고 `#E11D48` 계열의 배지, 테두리, 상단 강조선으로 정상 배송 흐름에서 벗어난 상태임을 표현했습니다. 지연은 “즉시 확인이 필요한 경고”, 반송은 “정상 배송 흐름에서 이탈한 상태”로 보고 시각적 성격을 분리했습니다.

### 레이아웃

PC에서는 2열 카드 레이아웃을 사용했습니다. 3열은 카드 폭이 좁아져 송장번호, 주소, 타임라인의 가독성이 떨어질 수 있고, 1열은 한 화면에서 확인 가능한 배송 건 수가 적다고 판단했습니다.

모바일에서는 1열 카드로 전환해 좁은 화면에서도 정보가 깨지지 않도록 했습니다. 상태 필터는 모바일에서도 기본적으로 한 화면에 노출되도록 구성하고, 더 좁은 화면에서는 3개씩 2줄로 정리되도록 했습니다.

### 필터와 검색 UX

상태 필터와 송장번호 검색은 독립 조건으로 동작하도록 설계했습니다. 특정 상태를 선택한 뒤 그 상태 안에서 송장번호를 검색할 수 있어 대량 배송 건 탐색에 유리합니다.

검색은 입력 즉시 실행하지 않고 검색 버튼 또는 Enter 입력으로 적용합니다. 하루 수백~수천 건의 데이터를 다루는 운영 환경을 고려해 불필요한 반복 필터링보다 명시적 검색 실행이 더 안정적이라고 판단했습니다. 검색 초기화는 검색어만 제거하고 현재 선택된 상태 필터는 유지합니다.

## 6. 엣지 케이스 대응

- 수령인 이름이 긴 경우 목록에서는 1줄 말줄임, 상세 모달에서는 전체 표시
- 출발지/도착지 주소가 긴 경우 목록에서는 최대 2줄 말줄임, 상세 모달에서는 전체 표시
- 단일 정보 필드가 비어 있는 경우 `-` 표시
- 예상 도착일이 없는 경우 목록과 상세 모달에서 `-` 표시
- 배송 이동 이력이 없는 경우 `배송 이동 이력이 없습니다.` 표시
- 메모가 없는 경우 `등록된 메모가 없습니다.` 표시
- mock 데이터 앞쪽에 고정 예외 데이터를 배치해 바로 확인 가능

## 7. 트레이드오프

현재 페이지네이션은 mock 데이터를 기준으로 클라이언트에서 처리했습니다. 실제 서비스에서는 배송 건 수가 많아질 수 있으므로, API와 연동할 경우 상태값, 송장번호, 페이지 정보를 서버에 전달해 서버 페이지네이션으로 확장하는 방식이 더 적합하다고 판단했습니다.

시간 제약상 아래 기능은 구현하지 않았습니다.

- 실제 API 연동
- 실시간 배송 상태 업데이트
- 서버 페이지네이션
- 스크롤이 길어졌을 때 필터와 검색 영역을 상단에 고정하는 처리
- 페이지 수가 많아졌을 때 원하는 페이지로 바로 이동하는 기능
- 담당자 배정, 알림, 카드별 액션 메뉴
- 실제 전화번호 마스킹 또는 권한별 개인정보 노출 제어
- 접근성 자동 테스트와 E2E 테스트

### 개선 방향

- 실제 API 응답 구조와 연결
- 서버 페이지네이션 적용
- 필터와 검색 영역을 상단에 고정해 긴 목록에서도 조건을 쉽게 변경할 수 있도록 개선
- 페이지 수가 많아질 경우 페이지 번호 생략 처리 또는 원하는 페이지로 바로 이동하는 기능 추가
- 지연 사유, 담당 센터, 처리 우선순위 등 운영 필터 추가
- 담당자 배정 및 내부 메모 수정 기능 추가
- 개인정보성 정보에 대한 권한별 노출 제어

## 8. 어려웠던 점

가장 고민했던 부분은 많은 배송 건을 한눈에 확인해야 하는 상황에서 카드 안에 어느 정도까지 정보를 보여줄지 결정하는 것이었습니다. 목록 형태가 많은 데이터를 빠르게 훑기에는 더 유리하지만, 과제 요구사항이 카드 컴포넌트였기 때문에 정보량과 가독성 사이의 균형을 잡는 것이 중요했습니다. 그래서 목록 카드에는 송장번호, 상태, 수령인, 출발지, 도착지, 예상 도착일, 진행 단계만 노출하고, 연락처나 상세 이력은 모달에서 확인하도록 분리했습니다.

검색과 필터 처리 방식도 고민했습니다. 실제 운영 환경에서는 배송 건이 수백~수천 건까지 늘어날 수 있기 때문에, 입력할 때마다 바로 검색되는 방식보다는 사용자가 송장번호를 입력한 뒤 검색 버튼이나 Enter로 실행하는 방식이 더 적절하다고 판단했습니다. 이번 과제에서는 mock 데이터를 기준으로 처리했지만, 실제 API와 연결될 경우 불필요한 요청을 줄일 수 있도록 상태값, 검색어, 페이지 정보를 분리해 관리했습니다.

또한 어느 정보까지 필수값으로 보고, 어떤 정보는 누락될 수 있다고 볼지도 고민했습니다. 송장번호, 수령인, 출발지, 도착지처럼 배송 건을 식별하는 정보는 기본적으로 존재한다고 보고 작업했지만, 발송인 연락처, 현재 위치, 배송 담당자, 이동 이력, 메모 등은 운영 상황에 따라 등록되지 않았을 수 있다고 판단했습니다. 그래서 단일 값은 `-`로 표시하고, 배송 이동 이력이나 메모처럼 영역 단위의 정보는 안내 문구로 처리했습니다.

실제 운영 환경에서는 연락처나 담당자 정보처럼 개인정보성이 있는 항목에 대해 권한별 노출 제어도 필요할 수 있다고 생각했습니다. 이번 과제 범위에서는 권한 처리를 구현하지 않았지만, 목록에는 노출하지 않고 상세 모달에서만 확인하도록 배치해 정보 노출 범위를 줄였습니다.
